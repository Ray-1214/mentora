## 建造順序
> 原則:照序執行,讓**任何時間點都有可 demo 的版本**。即使只做到階段 3,對競賽已是完整且有亮點的作品。

### 階段 1 — 修出題引擎【最優先・根本性】
解決「單字由上往下出 + 單調」。根因:`selectAnswerWords` 為完全決定性排序、零隨機。
- **1a 加權隨機抽樣** — 打散同條件單字的固定順序(止血,當天可驗證)。
- **1b 導入 SRS** — 間隔重複排程(FSRS 或 Leitner),依「該複習時機」出題,取代死板清單;沿用既有 `last_seen` / `consecutive_corrects`。
- **驗證**:測試腳本連跑選字 N 次,印出單字分布,證明不再照 DB 順序。
- 狀態:☐ 未開始

### 階段 2 — 自訂單字範圍 / 匯入
讓使用者自定字表與出題範圍(擴充既有 `VocabManager`)。
- 狀態:☐ 未開始

### 階段 3 — 錯題本強化 + 弱點路由
把「弱點分析 → 自動排進下次出題」做紮實(沿用 `getWeakGrammarPoints` / `getWeakVocabWords`)。這是「個人化家教」的核心賣點。
- 狀態:☐ 未開始

### 階段 4(Stretch)— 聽力(TTS)
用 TTS 把現有閱讀題型擴成聽力。**僅在階段 1–3 完成且有餘裕時做。**
> 註:Ray 在 #3 將「口說/發音評測」明確下放至 Backlog。「聽力」我暫放此處作 stretch(成本遠低於口說);若想完全延後,直接把本階段移到 Backlog。
- 狀態:☐ Stretch

## Backlog（非短期目標,以後考慮）
- **口說 / 發音評測**(STT + 發音評分)— 最重、最貴,延後。
- **手機版** — Capacitor 包同一份 React 核心,上 Android / iOS。
- **後端 API** — LLM 代理、帳號、token 記帳、廣告獎勵伺服器端驗證(SSV)。
- **看廣告換 token** — AdMob 獎勵式廣告 + 後端 SSV 驗證(前端發 token 會被作弊)。
- **上架後換較便宜 LLM**(如 DeepSeek)+ 單位經濟試算(廣告收入 vs 每次生成成本)。
- **移除硬編碼 API key**(`public/electron.js` fallback)— 上架前必做的安全前提。
